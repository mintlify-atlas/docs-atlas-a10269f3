## Welcome Contributors!

Thanks for your interest in contributing to Neru! We welcome contributions of all kinds — code, docs, bug reports, config examples, or ideas. Neru is a small project with an approachable codebase, making it a great place to get started with open source.

<Info>
  Before starting, please read our [Code of Conduct](https://github.com/y3owk1n/neru/blob/main/CODE_OF_CONDUCT.md). By participating, you agree to uphold it.
</Info>

## Getting Started

<Steps>
  <Step title="Search Existing Issues">
    Check if someone is already working on the same thing or if there's a related discussion:
    - [Open Issues](https://github.com/y3owk1n/neru/issues)
    - [Pull Requests](https://github.com/y3owk1n/neru/pulls)
  </Step>
  
  <Step title="Open an Issue First">
    For non-trivial changes, open an issue first to discuss the approach. This avoids wasted effort and helps align on implementation details before writing code.
  </Step>
  
  <Step title="Keep PRs Focused">
    Small, focused pull requests are preferred over large, sweeping changes. One logical change per PR makes review easier and faster.
  </Step>
</Steps>

## Making Changes

<Steps>
  <Step title="Fork and Clone">
    Fork the repository on GitHub and clone your fork:
    
    ```bash
    git clone https://github.com/YOUR_USERNAME/neru.git
    cd neru
    git remote add upstream https://github.com/y3owk1n/neru.git
    ```
  </Step>
  
  <Step title="Create a Branch">
    Create a branch from `main` with a descriptive name:
    
    ```bash
    git checkout -b feat/my-feature
    # or
    git checkout -b fix/bug-description
    ```
  </Step>
  
  <Step title="Make Changes">
    Follow the [Coding Standards](https://github.com/y3owk1n/neru/blob/main/docs/CODING_STANDARDS.md):
    - Write clear, readable code
    - Add godoc comments for exported symbols
    - Use proper error handling
    - Follow established naming patterns
  </Step>
  
  <Step title="Add Tests">
    All new code requires appropriate tests:
    - **Unit tests** for business logic
    - **Integration tests** for platform APIs
    
    See [Testing Guidelines](#testing-guidelines) below.
  </Step>
  
  <Step title="Run Pre-commit Checks">
    ```bash
    just fmt            # Format code
    just lint           # Run linters
    just test           # Run unit tests
    just build          # Verify build
    ```
  </Step>
  
  <Step title="Commit Changes">
    Use [conventional commits](#commit-messages):
    
    ```bash
    git add .
    git commit -m "feat: add grid subdivision mode"
    ```
  </Step>
  
  <Step title="Push and Open PR">
    ```bash
    git push origin feat/my-feature
    ```
    
    Open a pull request on GitHub with a clear description.
  </Step>
</Steps>

## Commit Messages

Neru uses [Conventional Commits](https://www.conventionalcommits.org/) to power automated releases via [Release Please](https://github.com/googleapis/release-please).

### Format

```
<type>(<optional scope>): <subject>

<optional body>

<optional footer>
```

### Types

| Type       | When to use                            | Examples                                |
| ---------- | -------------------------------------- | --------------------------------------- |
| `feat`     | New feature                            | `feat(grid): add recursive subdivision` |
| `fix`      | Bug fix                                | `fix(hints): correct overlay position`  |
| `docs`     | Documentation only                     | `docs: update configuration reference`  |
| `style`    | Formatting, no logic change            | `style: format with gofumpt`            |
| `refactor` | Code restructuring, no behavior change | `refactor: extract hint generation`     |
| `perf`     | Performance improvement                | `perf: cache accessibility elements`    |
| `test`     | Adding or updating tests               | `test: add grid calculation tests`      |
| `chore`    | Build, CI, dependencies, tooling       | `chore: update golangci-lint to v1.55`  |

### Examples

<Accordion title="Good commit message">
  ```
  feat(grid): add recursive subdivision mode
  
  Implement recursive grid subdivision allowing users to progressively
  narrow down screen areas. Includes:
  - New RecursiveGridMode implementation
  - Cell highlighting and backtracking
  - CLI command: neru grid --recursive
  
  Closes #123
  ```
</Accordion>

<Accordion title="Bad commit message">
  ```
  fix bug
  ```
  
  Too vague - doesn't describe what was fixed or why.
</Accordion>

## Architecture & Cross-Platform

Neru follows a **Hexagonal Architecture (Ports and Adapters)** pattern with strict platform isolation.

### Key Architectural Rules

<Warning>
  **The "One Rule"**: Non-darwin-tagged code must never import `internal/core/infra/platform/darwin`.
  
  This is enforced by `golangci-lint` using `depguard`.
</Warning>

<Steps>
  <Step title="Platform Isolation">
    OS-specific code must be strictly isolated. Use build tags:
    
    ```go
    //go:build darwin
    
    package darwin
    ```
  </Step>
  
  <Step title="Ports and Adapters">
    Define interfaces (Ports) in `internal/core/ports/` and implement them in `internal/core/infra/`.
  </Step>
  
  <Step title="Platform Factory">
    Use the factory pattern (`internal/core/infra/platform/factory.go`) to provide platform-specific implementations without darwin imports.
  </Step>
</Steps>

### Contributing to New Platforms

If you're working on Linux or Windows support:

1. Check the [Platform Status](https://github.com/y3owk1n/neru/blob/main/docs/ARCHITECTURE.md#platform-status)
2. Start by replacing `CodeNotSupported` stubs in `internal/core/infra/platform/<os>/`
3. Follow patterns from the macOS implementation
4. Add integration tests with `//go:build integration && <os>`

See the [Cross-Platform Guide](/advanced/cross-platform) for detailed porting instructions.

## Testing Guidelines

Neru separates tests into **unit tests** and **integration tests**:

| Type              | File Pattern            | Command                 | Build Tag     |
| ----------------- | ----------------------- | ----------------------- | ------------- |
| Unit tests        | `*_test.go`             | `just test`             | —             |
| Integration tests | `*_integration_test.go` | `just test-integration` | `integration` |

### Writing Tests

<Tabs>
  <Tab title="Unit Tests">
    Use mocks for external dependencies:
    
    ```go
    import "github.com/y3owk1n/neru/internal/core/ports/mocks"
    
    func TestHintService(t *testing.T) {
        mockAccessibility := mocks.NewMockAccessibilityPort()
        mockOverlay := mocks.NewMockOverlayPort()
        
        service := services.NewHintService(mockAccessibility, mockOverlay, ...)
        // Test service logic
    }
    ```
  </Tab>
  
  <Tab title="Integration Tests">
    Test with real platform APIs:
    
    ```go
    //go:build integration && darwin
    
    package accessibility
    
    func TestRealAccessibility(t *testing.T) {
        adapter := NewAdapter(...)
        elements, err := adapter.ClickableElements(ctx, filter)
        require.NoError(t, err)
        assert.NotEmpty(t, elements)
    }
    ```
  </Tab>
  
  <Tab title="Table-Driven Tests">
    Use table-driven tests for multiple test cases:
    
    ```go
    func TestGridCalculation(t *testing.T) {
        tests := []struct {
            name     string
            rows     int
            cols     int
            expected int
        }{
            {"2x2 grid", 2, 2, 4},
            {"3x3 grid", 3, 3, 9},
        }
        
        for _, tt := range tests {
            t.Run(tt.name, func(t *testing.T) {
                // Test logic
            })
        }
    }
    ```
  </Tab>
</Tabs>

### Test Coverage

```bash
# Run with coverage
go test -coverprofile=coverage.out ./...
go tool cover -html=coverage.out

# View coverage by package
go test -cover ./...
```

## Code Style

All code must follow the [Coding Standards](https://github.com/y3owk1n/neru/blob/main/docs/CODING_STANDARDS.md).

### Go Conventions

<Steps>
  <Step title="Imports">
    Use `goimports` for import organization:
    1. Standard library
    2. External packages
    3. Internal packages
  </Step>
  
  <Step title="Naming">
    - Use `camelCase` for unexported, `PascalCase` for exported
    - Use meaningful names: `accessibilityAdapter` not `aa`
    - Follow receiver conventions: short, consistent (e.g., `h *Handler`)
  </Step>
  
  <Step title="Error Handling">
    Use the custom error package:
    
    ```go
    import "github.com/y3owk1n/neru/internal/core/errors"
    
    return derrors.Wrap(err, "failed to get clickable elements")
    ```
  </Step>
  
  <Step title="Documentation">
    Add godoc comments for all exported symbols:
    
    ```go
    // AccessibilityPort defines the interface for platform accessibility APIs.
    // Implementations handle bridge complexity and live in internal/core/infra.
    type AccessibilityPort interface {
        // ...
    }
    ```
  </Step>
</Steps>

### Objective-C Guidelines

For Objective-C bridge code (`internal/core/infra/platform/darwin/`):

- Use descriptive names: `GetScreenBounds()` not `GetBounds()`
- Follow Apple naming conventions
- Properly manage memory with `CFRetain`/`CFRelease`
- Add comments explaining Cocoa API usage

### Formatting

```bash
# Format Go code
just fmt
# or manually
goimports -w .
gofumpt -w .

# Format Objective-C code
clang-format -i internal/core/infra/platform/darwin/*.m
```

## Pull Requests

### PR Title

Follow the same conventional commit format:

```
feat(hints): add multi-monitor support
fix(grid): correct cell boundary calculations
docs: update architecture guide
```

### PR Description

Include:

<Steps>
  <Step title="What Changed">
    Describe what you changed and why.
  </Step>
  
  <Step title="How to Test">
    Provide steps to test the changes.
  </Step>
  
  <Step title="Screenshots/Recordings">
    For UI changes, include screenshots or recordings.
  </Step>
  
  <Step title="Related Issues">
    Link related issues: `Closes #123` or `Relates to #456`.
  </Step>
  
  <Step title="Checklist">
    - [ ] Code follows [coding standards](https://github.com/y3owk1n/neru/blob/main/docs/CODING_STANDARDS.md)
    - [ ] Tests added/updated
    - [ ] Documentation updated
    - [ ] All CI checks pass
  </Step>
</Steps>

### PR Review Process

1. **Automated checks** run (lint, test, build)
2. **Maintainer review** (may request changes)
3. **Iterate** based on feedback
4. **Approval and merge** by maintainer

<Info>
  Be open to feedback and iterate. Reviews help improve code quality and ensure consistency.
</Info>

## Reporting Bugs

Open a [GitHub Issue](https://github.com/y3owk1n/neru/issues/new) with:

<Steps>
  <Step title="Environment">
    - macOS/Linux/Windows version
    - Neru version (`neru version`)
    - Go version (`go version`)
  </Step>
  
  <Step title="Steps to Reproduce">
    Provide minimal, specific steps:
    1. Run `neru hints`
    2. Press `a`
    3. Observe incorrect behavior
  </Step>
  
  <Step title="Expected vs Actual">
    - **Expected**: Element should be clicked
    - **Actual**: Nothing happens
  </Step>
  
  <Step title="Logs">
    Enable debug logging and attach relevant lines:
    ```toml
    [logging]
    log_level = "debug"
    ```
    
    ```bash
    # macOS
    tail -f ~/Library/Logs/neru/app.log
    ```
  </Step>
  
  <Step title="Screenshots/Recordings">
    For visual issues, include screenshots or recordings.
  </Step>
</Steps>

## Feature Requests

Open a [GitHub Issue](https://github.com/y3owk1n/neru/issues/new) or start a [Discussion](https://github.com/y3owk1n/neru/discussions) describing:

- **What** you'd like to see
- **Why** it would be useful (your use case)
- **How** you envision it working (optional but helpful)

## Community Guidelines

<Steps>
  <Step title="Be Respectful">
    Treat all contributors with respect. We're all here to improve Neru.
  </Step>
  
  <Step title="Be Patient">
    Maintainers are volunteers. Response times may vary.
  </Step>
  
  <Step title="Be Constructive">
    Provide actionable feedback. "This doesn't work" is less helpful than "When I do X, Y happens instead of Z."
  </Step>
  
  <Step title="Be Open">
    Be open to feedback on your contributions. Reviews help everyone learn.
  </Step>
</Steps>
