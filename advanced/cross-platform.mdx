## Platform Status

Neru is designed for cross-platform support with macOS as the primary platform. Linux and Windows support is actively being developed.

## Platform Architecture

### Design Principles

Neru follows **Hexagonal Architecture (Ports and Adapters)** for clean platform separation:

<Steps>
  <Step title="Shared Business Logic">
    All core logic (hint generation, grid calculations, mode transitions) is written in pure Go and resides in `internal/core/domain` and `internal/app/services`.
  </Step>
  
  <Step title="Platform Isolation">
    OS-specific code is strictly isolated. Non-darwin code must never import macOS-specific packages.
  </Step>
  
  <Step title="Ports and Adapters">
    System capabilities (Accessibility, Hotkeys, Overlays) are defined as interfaces (Ports) in `internal/core/ports`. Concrete implementations (Adapters) reside in `internal/core/infra`.
  </Step>
  
  <Step title="Build Tag Separation">
    OS-specific files use Go build tags (e.g., `//go:build darwin`) to ensure they are only compiled for the target platform.
  </Step>
</Steps>

### The "One Rule"

<Warning>
  **Non-darwin-tagged code must never import `internal/core/infra/platform/darwin`.**
  
  This rule is enforced by `golangci-lint` using `depguard`.
</Warning>

## Porting Guide

### Adding a Feature to an Existing Platform

<Steps>
  <Step title="Find the Right Adapter">
    Locate the adapter responsible for the feature:
    
    | Feature                          | Location                                          |
    | -------------------------------- | ------------------------------------------------- |
    | Screen bounds, cursor, dark mode | `internal/core/infra/platform/<os>/system.go`     |
    | Global hotkeys                   | `internal/core/infra/hotkeys/manager_<os>.go`     |
    | Global keyboard event tap        | `internal/core/infra/eventtap/eventtap_<os>.go`   |
    | Application watcher              | `internal/core/infra/appwatcher/platform_<os>.go` |
    | UI overlays                      | `internal/app/components/*/overlay_<os>.go`       |
  </Step>
  
  <Step title="Replace the Stub">
    Most unimplemented methods currently return:
    
    ```go
    return derrors.New(derrors.CodeNotSupported, "X not yet implemented on linux")
    ```
    
    Replace that with a real implementation. Remove the `TODO` comment when done.
  </Step>
  
  <Step title="Add Tests">
    Unit tests go next to the implementation file. Integration tests (that require a real display/OS) go in `*_integration_<os>_test.go` files with the matching build tag:
    
    ```go
    //go:build integration && linux
    ```
  </Step>
  
  <Step title="Run CI Locally">
    ```bash
    # On the target OS:
    just lint
    just vet
    just test
    just build
    ```
  </Step>
</Steps>

### Adding a New OS-Specific Operation

#### Option A: Add to SystemPort

For broadly applicable operations:

<Steps>
  <Step title="Add Method to Port">
    Add the method to `internal/core/ports/system.go`:
    
    ```go
    type SystemPort interface {
        // ... existing methods
        NewOperation(ctx context.Context) error
    }
    ```
  </Step>
  
  <Step title="Implement for macOS">
    Implement in `internal/core/infra/platform/darwin/system.go`:
    
    ```go
    func (s *SystemAdapter) NewOperation(ctx context.Context) error {
        // Real implementation using macOS APIs
    }
    ```
  </Step>
  
  <Step title="Add Stubs for Other Platforms">
    Add `CodeNotSupported` stubs in:
    - `internal/core/infra/platform/linux/system.go`
    - `internal/core/infra/platform/windows/system.go`
    
    ```go
    func (s *SystemAdapter) NewOperation(ctx context.Context) error {
        return derrors.New(derrors.CodeNotSupported, "NewOperation not yet implemented on linux")
    }
    ```
  </Step>
  
  <Step title="Inject SystemPort">
    Inject `SystemPort` into the service/handler that needs it via `internal/app/app_initialization.go`.
  </Step>
</Steps>

#### Option B: Use a Dispatch Pair

For isolated operations needed in one infra package:

<Steps>
  <Step title="Create Darwin Implementation">
    Create `internal/core/infra/<package>/platform_darwin.go`:
    
    ```go
    //go:build darwin
    package <package>
    
    import "github.com/y3owk1n/neru/internal/core/infra/platform/darwin"
    
    func platformDoThing() {
        darwin.DoThing()
    }
    ```
  </Step>
  
  <Step title="Create Stub for Other Platforms">
    Create `internal/core/infra/<package>/platform_stub.go`:
    
    ```go
    //go:build !darwin
    package <package>
    
    func platformDoThing() {
        // No-op or return error
    }
    ```
  </Step>
  
  <Step title="Call from Shared Code">
    Call `platformDoThing()` from the package's shared code (no build tag needed).
  </Step>
</Steps>

### Platform Factory Pattern

The platform factory (`internal/core/infra/platform/factory.go`) provides the correct `SystemPort` implementation without polluting shared code:

**Shared interface** (`factory.go`):
```go
package platform

// NewSystemPort is defined in build-tagged files
```

**Darwin implementation** (`factory_darwin.go`):
```go
//go:build darwin

package platform

import "github.com/y3owk1n/neru/internal/core/infra/platform/darwin"

func NewSystemPort() (ports.SystemPort, error) {
    return darwin.NewSystemAdapter(), nil
}
```

**Linux implementation** (`factory_linux.go`):
```go
//go:build linux

package platform

import "github.com/y3owk1n/neru/internal/core/infra/platform/linux"

func NewSystemPort() (ports.SystemPort, error) {
    return linux.NewSystemAdapter(), nil
}
```

## Platform-Specific Requirements

### Linux

<AccordionGroup>
  <Accordion title="Accessibility (AT-SPI)">
    Linux accessibility requires **AT-SPI (Assistive Technology Service Provider Interface)**.
    
    **Required packages**:
    - `at-spi2-core`
    - `at-spi2-atk`
    
    **Key APIs**:
    - `atspi_get_desktop()` - Get desktop root
    - `atspi_accessible_get_child_at_index()` - Traverse UI tree
    - `atspi_component_get_extents()` - Get element bounds
    
    **Implementation location**: `internal/core/infra/platform/linux/accessibility.go`
  </Accordion>
  
  <Accordion title="Keyboard Events (X11/Wayland)">
    Linux keyboard interception depends on the display server:
    
    **X11**:
    - Use `XGrabKey()` for global hotkeys
    - Use `XRecordExtension` for event tap
    
    **Wayland**:
    - No direct API for global hotkeys
    - Requires compositor support (e.g., via D-Bus)
    
    **Implementation location**: `internal/core/infra/eventtap/eventtap_linux.go`
  </Accordion>
  
  <Accordion title="Overlays (GTK/Qt)">
    Linux overlays can use:
    - **GTK 3/4**: `gtk_window_new()` with transparency
    - **Qt**: `QWidget` with `Qt::FramelessWindowHint`
    
    **Implementation location**: `internal/app/components/overlayutil/factory_linux.go`
  </Accordion>
</AccordionGroup>

### Windows

<AccordionGroup>
  <Accordion title="Accessibility (UI Automation)">
    Windows accessibility requires **UI Automation (UIA)**.
    
    **Required libraries**:
    - `UIAutomationCore.h`
    - `UIAutomationClient.h`
    
    **Key APIs**:
    - `IUIAutomation::GetRootElement()` - Get desktop root
    - `IUIAutomationElement::FindAll()` - Query elements
    - `IUIAutomationElement::GetBoundingRectangle()` - Get element bounds
    
    **Implementation location**: `internal/core/infra/platform/windows/accessibility.go`
  </Accordion>
  
  <Accordion title="Keyboard Events (Windows Hooks)">
    Windows keyboard interception uses **Windows Hooks**.
    
    **Key APIs**:
    - `RegisterHotKey()` - Register global hotkeys
    - `SetWindowsHookEx(WH_KEYBOARD_LL)` - Low-level keyboard hook
    - `CallNextHookEx()` - Pass events to next hook
    
    **Implementation location**: `internal/core/infra/eventtap/eventtap_windows.go`
  </Accordion>
  
  <Accordion title="Overlays (Win32)">
    Windows overlays use **Win32 API**.
    
    **Key APIs**:
    - `CreateWindowEx()` with `WS_EX_LAYERED | WS_EX_TRANSPARENT`
    - `SetLayeredWindowAttributes()` - Set transparency
    - `UpdateLayeredWindow()` - Update contents
    
    **Implementation location**: `internal/app/components/overlayutil/factory_windows.go`
  </Accordion>
</AccordionGroup>

## CLI Layer Cross-Platform Notes

### Service Management

`internal/cli/services.go` carries `//go:build darwin` because it uses `launchctl` and macOS `.plist` files.

<Steps>
  <Step title="macOS (launchctl)">
    Uses Launch Agents for daemon management:
    - Install: Copy `.plist` to `~/Library/LaunchAgents/`
    - Start: `launchctl load`
    - Stop: `launchctl unload`
  </Step>
  
  <Step title="Linux (systemd)">
    To add Linux service management:
    1. Create `internal/cli/services_linux.go` with `//go:build linux`
    2. Implement install/uninstall/start/stop using `systemctl`
    3. Register subcommands in `init()`
  </Step>
  
  <Step title="Windows (Task Scheduler)">
    To add Windows service management:
    1. Create `internal/cli/services_windows.go` with `//go:build windows`
    2. Use Task Scheduler API or `schtasks.exe`
    3. Register subcommands in `init()`
  </Step>
</Steps>

## Application Identifier Terminology

The codebase uses "bundle ID" as a generic term for the platform application identifier:

| Platform | Term                        | Example                          |
| -------- | --------------------------- | -------------------------------- |
| macOS    | Bundle ID                   | `com.apple.Safari`               |
| Linux    | Desktop ID / executable     | `firefox.desktop` or `firefox`   |
| Windows  | AppUserModelID / executable | `Microsoft.Edge` or `msedge.exe` |

<Info>
  The `FocusedAppBundleID` method in `ports.AccessibilityPort` returns whatever the platform uses. The exclusion list in config (`general.excluded_apps`) should use the same format.
</Info>

## Contributing Platform Support

### Prerequisites

<Tabs>
  <Tab title="macOS">
    - Xcode Command Line Tools: `xcode-select --install`
    - CGo enabled: `CGO_ENABLED=1`
    - Access to macOS APIs
  </Tab>
  
  <Tab title="Linux">
    - `gcc` and `pkg-config`
    - AT-SPI development libraries: `sudo apt install libatspi2.0-dev`
    - X11 development libraries: `sudo apt install libx11-dev libxtst-dev`
    - GTK development libraries: `sudo apt install libgtk-3-dev`
  </Tab>
  
  <Tab title="Windows">
    - C compiler (MinGW or MSVC)
    - Windows SDK with UI Automation headers
    - CGo enabled: `CGO_ENABLED=1`
  </Tab>
</Tabs>

### Development Workflow

<Steps>
  <Step title="Choose a Capability">
    Pick an unimplemented capability from the [Platform Status](#platform-status) table.
  </Step>
  
  <Step title="Implement the Adapter">
    Follow the [Porting Guide](#porting-guide) to implement the feature for your platform.
  </Step>
  
  <Step title="Add Tests">
    Write unit and integration tests:
    - Unit tests: `*_test.go`
    - Integration tests: `*_integration_<os>_test.go` with `//go:build integration && <os>`
  </Step>
  
  <Step title="Test Locally">
    ```bash
    just lint    # Run linters
    just test    # Run unit tests
    just test-integration  # Run integration tests
    just build   # Build binary
    ```
  </Step>
  
  <Step title="Submit PR">
    Open a pull request with:
    - Implementation details
    - Test coverage
    - Platform-specific notes
  </Step>
</Steps>

## Testing Cross-Platform Code

### CI/CD Platform Support

GitHub Actions runs tests on all platforms:

```yaml
strategy:
  matrix:
    os: [macos-latest, ubuntu-latest, windows-latest]
    go-version: [1.26]
```

### Running Platform-Specific Tests

```bash
# Run all tests
just test

# Run only integration tests (requires real OS APIs)
just test-integration

# Run with race detection
just test-race
```
